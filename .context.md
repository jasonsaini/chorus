# Chorus — Project Context

## What it is
Chorus is a real-time collaborative chat app where multiple users share a single AI context window. Everyone in a room sees the same conversation, including Claude's responses. The core idea: vibe code and ideate on projects together with AI as a participant.

## Architecture

**Backend**: Spring Boot 3.4 / Java 21, deployed on Railway
- `config/` — Spring config and @ConfigurationProperties (ChorusProperties)
- `domain/` — Immutable domain objects (Room, ChatMessage — no JPA)
- `service/` — Business logic: RoomService (DB), ChatAiService (Claude), AiReplyService (async), GitHubService (OAuth + API)
- `ws/` — STOMP WebSocket controllers and SessionRegistry
- `web/` — REST controllers and DTOs
- `persistence/` — JPA entities and repositories

**Frontend**: React + TypeScript, deployed on Netlify
- Config: `VITE_API_ORIGIN` env var points at the backend

**Database**: H2 file-backed (Railway volume) — rooms, messages, participants persist across restarts

## Key flows

**Chat**: User sends STOMP message → `ChatStompController` validates session → persists USER message → publishes to `/topic/room/{roomId}` → if AI triggered, `AiReplyService.replyAsync()` calls Claude → publishes AI reply to same topic

**AI trigger modes**: `always` (every message) or `mention` (only when message contains `@claude`)

**Context window**: `RoomService.conversationForModel()` applies a token budget sliding window (default 8000 tokens, ~chars/4 estimate) — drops oldest messages first

**GitHub OAuth**: `GET /api/auth/github?roomId=X` → GitHub OAuth flow → token stored per room → `POST /api/rooms/{roomId}/repo` links a repo → Claude gets `.context.md` injected as system context on every message

## Current state
- Multi-user real-time chat with Claude working in production
- GitHub OAuth implemented — users can link a repo to a room
- Repo context injection: looks for `.context.md` at repo root, injected as system message before conversation history
- WebSocket heartbeat configured (25s) to prevent Railway dropping connections
- No auth — anyone can join any room by roomId (intentional for MVP)

## Stack decisions worth knowing
- H2 over Postgres: simplicity for MVP, Railway volume keeps data persistent
- Simple broker over full message broker: no RabbitMQ/Redis dependency, fine for current scale
- Token budget is approximate (char-based estimate, not real tokenizer) — intentional, avoids jtokkit dependency in hot path
- GitHub tokens stored in plaintext in DB — fine for MVP, needs encryption before public launch
